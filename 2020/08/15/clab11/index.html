CS61B Clab11 solution

Posted on 2020-08-15 In CS61B Valine:

恩最后一个challenge lab了，话不多说，内容如下：

Clab11

Introduction
The hot new Cal startup AirBearsnBeds has hired you to create an algorithm to help them place their customers in the best possible homes to improve their experience. They are currently in their alpha stage so their only customers (for now) are bears. Now, a little known fact about bears is that they are very, very picky about their bed sizes: they do not like their beds too big or too little - they like them just right. Bears are also sensitive creatures who don’t like being compared to other bears, but they are perfectly fine with trying out beds.

The Problem
Given a list of Bears with unique but unknown sizes and a list of Beds with corresponding but also unknown sizes (not necessarily in the same order), return a list of Bears and a list of Beds such that that the ith Bear in your returned list of Bears is the same size as the ith Bed in your returned list of Beds. Bears can only be compared to Beds and we can get feedback on if the Bed is too large, too small, or just right. In addition, Beds can only be compared to Bears and we can get feedback if the Bear is too large for it, too small for it, or just right for it.

Constraints
Your algorithm should run in O(NlogN) time. It may be helpful to figure out the naive O(N^2) solution first and then work from there.

Provided Files
Bear.java : A Bear object that is Comparable to Beds.
Bed.java : A Bed object that is Comparable to Bears.
HiddenComparable.java : Used to implement Bears and Beds. You should not need to use this file or its method at all.
Pair.java : A utility class that represents a two-item tuple. This is useful if you want to return two items in a function call. You don’t have to use this but you are free to.
BnBSolver.java : Your solution to this problem should be implemented in here. Do not change the provided method headers, but you may add extra methods and attributes.
BnBSolverTests.java : Provided tests.
BnBSolverTimingTests.java : Visualizes the runtime of your solver compared to other algorithms that run in Θ(N^2)and Θ(Nlog⁡N) time.

Solution

（1）嗯，最容易想到的想法就是selection了 ：我们保持bears序列不动，创造一个和bears序列一样的新beds序列 ：对i号位置的bears，只要在原bed序列中iterate一遍找到和它size相对的bed放入新序列的i号位置即可。这样，时间复杂度就是O(n^2), 也就是josh hug说的naive solution。

(2) 仔细考虑一下问题和排序的区别：排序是给定序列x，把x排成序列y(其中序列y是递增序列），现在的问题看上去就是给定序列x，把x排成序列y，其中y是任意给定的序列。但是认真考虑一下就知道如果按这个思路，是不可能以O(NlogN)的复杂度完成任务的，因为在排序的时候，由于y满足递增，我们只需要让x向递增方向靠拢即可，也就是根本上不需要y序列，只需要考察x序列自身的inversion情况即可，但是对于任意给定的序列y，我们要对x排序必须要知道y序列的情况，又因为y是乱序的，每次考察y序列都必须iterate一遍，那么不可能避免O(N^2)的复杂度。因此直接不用考虑这样的思路了。

(3) 接下来非常自然的思路就是把x序列和y序列都排序得到新x序列和新y序列，因为两个序列是一样的，自然排序之后满足第i号位置的size相同。很容易想到，必须依靠x序列对y序列进行排序(因为y序列里元素不能比较)；我们考虑一下几种常见的排序: selection sort, heap sort, insertion sort, merge sort, quick sort….满足O(nlogn)的只有heap sort, merge sort和quick sort。heap sort实际上就是一种selection sort，必须要维护heap structure, 在这里显然是不可能的，因此直接不予考虑。接下来考虑merge sort的思路，从长序列转化成短序列，会发现无法做到对应（举例而言，beds长序列是 b1→b2，bears长序列是e1→e2，将他们转化成两组短序列 (beds短序列b1, bears短序列e1) (beds短序列b2，bears短序列e2)的时候会发现很难做到b1和e1里的数是相对应的，因此放弃；

(4)接下来就是quick sort的思路，我们取bears序列中的第一个数pivot，这样可以把beds序列partition成比pivot小的beds，和pivot相等的beds，比pivot大的beds，然后我们继续partition不就成了嘛，看起来有戏！我很开心，但是立即意识到一个问题，接下来怎么办？也就是说, 在quick sort的时候，我们partition完了，只需在smallBeds/largeBeds里取某个数再分就可以recursion了，但是在这里，我们必须依靠bears序列里的数来对beds进行partition, 怎么做到接下来与smallBeds/largeBeds对应的两个bears序列里面包含的数是一样的呢？我们刚刚用bears里的pivot把Beds分成了比这个pivot小、大和相等的三组序列，那么，我们再用与这个pivot相等的Beds序列里的数去把bears分成smallBears, equalBears和greaterBears然后smallBeds和smallBears相对，equalBeds和equalBears相对，greaterBeds和greaterBears相对不就行了嘛~
